# A09 – Logging & Alerting Failures: Solutions

## Challenge 1 – Password in Logs
1. Go to `/a09/login-bad`, login with `alice` / `sunshine` (success) and then wrong password (failure).
2. Navigate to `/a09/view-logs`.
3. In `bad_app.log` you'll see:
   ```
   2025-01-12 10:00:00 LOGIN ATTEMPT user=alice password=sunshine ip=127.0.0.1
   2025-01-12 10:00:01 LOGIN SUCCESS user=alice
   ```

**Fix:** Never log credentials:
```python
# BAD
logger.debug(f"LOGIN ATTEMPT user={username} password={password}")

# GOOD — log user identity only
logger.info(f"AUTH_SUCCESS username={username} ip={ip}")
logger.warning(f"AUTH_FAILURE username={username} ip={ip}")
```

## Challenge 2 – Silent Failures
The `bad_app.log` has no entry for failed logins (only successes logged).
An attacker can brute-force indefinitely with no trace in logs.

In `good_app.log`, every failure generates `AUTH_FAILURE` at WARNING level.

## Challenge 3 & 4 – Large Transfer / Card in Logs
Submit $9,999 transfer with card `4111111111111111` at `/a09/transfer-bad`.
In `bad_app.log`:
```
TRANSFER user=alice amount=9999.0 card=4111111111111111
```
Full PAN (Primary Account Number) stored in logs — PCI-DSS violation.

At `/a09/transfer-good`, $9,999 triggers a WARNING anomaly alert, and the log shows:
```
TRANSFER user=alice amount=9999.0 card=**** **** **** 1111
```

**Fix:**
- Mask card: store only last 4 digits
- Alert on: large amounts, off-hours, unusual patterns
- Use structured logging (JSON) for SIEM integration
